# Abyss Echoes 全量重设计 v2

## 1. 目标

把 Abyss Echoes 从当前“传统关卡推进 + 英雄长期升级 + 大量装备管理”的雏形，重构为一个更适合 CLI 形态、节奏更清晰、决策更集中的 **run-based 小队自动战斗构筑游戏**。

这次不是局部调参，而是 **方向级重做**：

1. 重新定义玩家每 3-10 分钟内的核心决策。
2. 重新定义成长来源。
3. 重新定义内容组织方式。
4. 保留现有可复用的技术骨架：Python、CLI、三栏 UI、自动战斗。

## 2. 为什么要推翻现有内容

## 2.1 当前方向的问题

当前设计的问题不是“功能不够多”，而是“乐趣重心不够聚焦”。

主要问题：

1. **成长来源过散**
   - 英雄等级
   - 装备评分
   - 词缀比较
   - 专精
   - 技能等级
   - 可能还要加 milestone / augment / capstone

   这些层都能做，但放在 CLI 原型里会把注意力切碎。

2. **装备系统信息负担过高**
   - 多槽位
   - 词缀比较
   - 背包筛选
   - 推荐结论

   这些更像 ARPG 背包管理，而不是终端原型最该突出的乐趣。

3. **线性关卡推进太平**
   - 下一关、开打、掉数值、再下一关
   - 缺少路线选择与风险判断
   - 玩家很难产生“这一 run 我走成了什么构筑”的记忆点

4. **英雄差异存在，但成长路径不够鲜明**
   - 角色身份有了
   - 但一局里形成 build identity 的过程仍然偏弱

5. **CLI 优势没有被真正利用**
   - CLI 最适合：清楚的状态、明确的选择、短文本高频反馈
   - CLI 不适合：长期堆积、反复翻背包、海量低价值属性噪音

## 2.2 结论

因此，这次不再继续给旧框架补系统，而是直接重定游戏主循环：

> 从“长期养成驱动”转向“单次 run 构筑驱动”。

## 3. 新北极星

新的产品一句话定义：

> **一个 5 人小队的深渊远征 CLI 自动战斗游戏：玩家在一轮轮分叉路径中选择战斗、事件与构筑奖励，逐步把队伍塑造成一套成型的回响流派。**

玩家要记住的，不再是：
- 我背包里有多少件 73 分和 81 分的手套

而是：
- 这一局我走的是“燃烧扩散队”
- 我为了赌高风险契约放弃了恢复节点
- 我在第 4 层把辅助转成了充能核心
- 我这局后排成型很快，但前排扛不住 boss 第二轮爆发

## 4. 重设计后的产品支柱

## 4.1 支柱 A：每一局都要有构筑方向
每次 run 必须形成一个可命名的方向，而不是纯数值上涨。

示例：
- 灼烧扩散
- 高频普攻
- 终结收割
- 护盾反击
- 充能爆发
- 减益压制

## 4.2 支柱 B：决策必须短、硬、清晰
高频决策应该是：
- 三选一奖励
- 路线分叉
- 风险换收益
- 角色强化分支

而不是：
- 看 8 行词缀差异
- 来回切 9 个装备槽位
- 处理大量收益极小的背包更替

## 4.3 支柱 C：队伍协同高于单体堆数值
重点不再是单英雄装备评分，而是整队联动：
- 谁提供点火
- 谁扩散 debuff
- 谁补能量
- 谁负责开场承伤
- 谁负责斩杀收尾

## 4.4 支柱 D：失败必须提供可复盘的原因
失败后，玩家应该能读懂：
- 前排崩得太快
- 没有净化
- 伤害成型太慢
- 路线贪心导致恢复不足
- 选到的构筑件彼此不配合

## 5. 保留项与推翻项

## 5.1 保留项
以下内容继续保留：

1. Python CLI 技术路线
2. 三栏终端 UI 架构
3. 自动战斗基础形态
4. 5 人队伍
5. 速度 / 能量 / 自动施法 这一套核心战斗骨架
6. 深渊、回响、遗迹这一类世界观方向
7. 面向 AI 可读 spec 驱动开发的方式

## 5.2 推翻项
以下内容不再作为核心方向：

1. **长期英雄等级** 作为主成长来源
2. **大背包 + 多装备槽 + 词缀比较** 作为主决策来源
3. **线性下一关推进** 作为主结构
4. **大量常驻数值积累** 作为主要快感来源
5. **固定英雄 roster 的长期培养** 作为第一优先级体验

## 6. 新核心循环

## 6.1 单次 run 循环

```text
进入深渊 run
-> 选择初始 5 人小队
-> 选择初始回响核心 / 初始遗物
-> 在地图节点间推进
-> 战斗 / 事件 / 恢复 / 锻造 / 招募 / 精英 / Boss
-> 每场后获取三选一成长
-> 队伍在 run 中逐渐成型
-> 击败层级 Boss 或 run 失败
-> 结算并解锁新内容
-> 开启下一轮 run
```

## 6.2 局外循环
局外保留“轻 meta”，但不做重数值养成。

局外主要解锁：
- 新英雄
- 新回响核心池
- 新遗物池
- 新事件
- 新风险契约
- 新 boss 词条

局外不主打：
- 英雄永久等级
- 永久刷装备评分
- 复杂材料树

## 7. 新地图与推进结构

## 7.1 地图结构
每次 run 使用 8-12 个节点的分叉地图。

节点类型：
- `battle`：普通战斗
- `elite`：精英战斗
- `event`：事件抉择
- `rest`：恢复 / 复活 / 清除负面状态
- `forge`：升级一个构筑件或替换一个旧构筑件
- `recruit`：替换或补充队伍成员
- `boss`：层级 Boss

## 7.2 设计目的
地图分叉的意义是让玩家做路线决策：
- 稳定推进
- 贪构筑强度
- 赌事件收益
- 补前排生存
- 提前打精英换高质量奖励

## 7.3 推荐 run 节奏
推荐节奏：
- 第 1 层：建立方向
- 第 2 层：补齐协同
- 第 3 层：成型与 boss 检验

每层 1 个 boss。
完整 MVP run 建议为 3 层结构。

## 8. 新成长系统：用 run 内构筑替代长期等级

## 8.1 成长层级总览
新成长系统分为四层：

1. **回响核心（Echo Core）**
   - 定义整队方向
   - 开局选 1

2. **回响片段（Echo Fragment）**
   - 场后奖励三选一
   - 提供流派增幅

3. **角色觉醒（Awakening）**
   - 英雄在 run 中到达节点时获得二选一强化
   - 强化英雄身份

4. **遗物（Relic）**
   - 稀有的全队规则修改器
   - 数量少，但影响大

## 8.2 删除旧成长核心
以下旧成长核心删除或降级为非核心：
- 英雄经验条
- 每级属性成长
- 多段技能 rank 追踪
- 大量装备评分对比

这些不是不能存在，而是不再是最主要的玩家关注点。

## 9. 回响核心系统

## 9.1 定义
回响核心是每局开头选的“流派底座”。

每个核心定义：
- 它鼓励什么样的队伍行为
- 它会让哪类奖励更强
- 它会把哪些英雄变成核心受益者

## 9.2 MVP 核心池
推荐首批 6 个：

1. `embers`
   - 围绕燃烧 / DOT / 爆炸
2. `execution`
   - 围绕低血量收割 / 连斩 / 终结
3. `bulwark`
   - 围绕护盾 / 反击 / 前排承压
4. `overclock`
   - 围绕速度 / 普攻频率 / 行动条
5. `conduit`
   - 围绕能量 / 主动技循环
6. `hexweave`
   - 围绕减益层数 / 控制 / 易伤

## 9.3 设计规则
回响核心必须：
- 在 1 句话内能说清方向
- 能影响整个 run 的奖励价值判断
- 能与多名英雄形成协同
- 不要求玩家阅读大段规则文本才能理解

## 10. 回响片段系统

## 10.1 定义
回响片段是战后高频奖励，构成 run 内 build 的主体。

## 10.2 奖励规则
每场战斗结束后，通常给 3 选 1：
- 1 个偏当前核心的片段
- 1 个通用片段
- 1 个高风险 / 低配合度片段

## 10.3 片段类型
片段类型示例：
- `keyword_boost`
- `role_boost`
- `resource_boost`
- `conditional_trigger`
- `survival_patch`
- `finisher`

## 10.4 示例
```yaml
echo_fragment:
  id: embers_chain
  family: embers
  effect: 燃烧目标死亡时，将 50% 剩余燃烧层数扩散给随机敌人
  tags: [burn, spread, snowball]
```

```yaml
echo_fragment:
  id: emergency_barrier
  family: neutral
  effect: 我方后排首次跌破 40% 生命时，获得护盾
  tags: [defense, backline]
```

## 11. 角色觉醒系统

## 11.1 定义
英雄不再靠长期升级成长，而是在 run 内经历固定成长节点。

## 11.2 触发节点
推荐在以下时点获得觉醒：
- 第 2 场后
- 第 5 场后

每个英雄每局最多 2 次觉醒。

## 11.3 选择规则
每次觉醒提供二选一。

示例：
```yaml
awakening_choice:
  hero_id: sacred_priest
  options:
    - id: grace_pulse
      effect: 主动治疗后额外为最低生命友军提供小护盾
    - id: hymn_charge
      effect: 治疗暴击时额外回复能量
```

## 11.4 设计意义
这样保留“角色成长感”，但避免长期等级树带来的 UI 负担和调参成本。

## 12. 遗物系统

## 12.1 定义
遗物是低频、高影响的全队 modifier。

## 12.2 获取方式
主要来自：
- 精英奖励
- 事件
- Boss 奖励

## 12.3 设计约束
遗物必须：
- 少而强
- 易读
- 改变玩法判断
- 不制造长篇规则依赖

示例：
- `灰烬圣匣`：所有燃烧伤害降低 20%，但触发爆炸时造成范围溅射
- `逆时砂盘`：每第 5 次我方行动后，行动条最低的友军立即获得推进
- `黑潮契印`：精英战奖励 +1 选项，但战斗开始时全队生命 -15%

## 13. 英雄内容重组

## 13.1 英雄数量方向
当前不建议继续维持“大而散”的英雄池。

MVP 重设计建议：
- 首发 **8 名英雄**
- 每名英雄的身份必须非常鲜明
- 允许未来扩到 10-12，但不是首要目标

## 13.2 角色设计原则
每名英雄必须清晰回答：
1. 他在队伍里负责什么
2. 他最喜欢什么回响方向
3. 他最怕什么短板
4. 他通过什么觉醒分支改变打法

## 13.3 推荐首发 8 人结构
- 2 坦克
- 2 近战输出
- 2 远程/术士输出
- 2 辅助

这样能保证：
- 阵容组合够用
- 内容量可控
- 每个英雄都能被认真打磨

## 14. 装备系统重做

## 14.1 总方向
当前装备系统从“核心玩法”降级为“辅助成长层”。

## 14.2 删除项
删除以下重负担设计：
- 大量通用掉落
- 多槽位频繁替换
- 词缀海
- 长期共享背包管理作为主体验

## 14.3 替代方案
重做后采用更轻量的装备/道具表达，二选一：

### 方案 A：完全取消传统装备栏
改为：
- 回响片段
- 遗物
- 角色觉醒

这是最激进、最干净的方案。

### 方案 B：保留极简装备
仅保留：
- `weapon`
- `trinket`

规则：
- 每英雄最多 2 槽
- 不做复杂随机词缀
- 物品更像 build piece，而不是评分材料

## 14.4 当前建议
默认采用 **方案 B**：
- 这样还能保留“拿到物品”的满足感
- 但不会重新掉回背包管理泥潭

## 15. 战斗系统调整原则

## 15.1 保留基础骨架
战斗底层继续保留：
- 自动战斗
- 速度驱动行动条
- 能量满后释放主动技
- 前后排 / 目标优先级

## 15.2 调整重点
需要调整的是“战斗的 build 接口”，不是全部推翻引擎。

重点改这些层：
- 回响片段怎样挂到战斗事件上
- 遗物怎样介入 battle hooks
- 英雄觉醒怎样改写技能或被动
- Boss 怎样对 build 提出检验

## 15.3 Boss 设计原则
Boss 必须是“构筑考试题”，而不是单纯大号木桩。

Boss 应至少测试以下其中 2 项：
- 前排承压
- 后排保护
- 净化/驱散
- 爆发窗口
- 持久战资源循环
- 群体处理能力

## 16. UI 映射原则

本次虽然重做玩法，但继续沿用 `cli-ui-optimization.md` 的表现哲学。

## 16.1 左栏
左栏继续负责：
- run 状态
- 当前层/节点
- 队伍状态
- 风险概览

## 16.2 中栏
中栏展示当前主内容：
- 地图路径
- 奖励三选一
- 事件抉择
- 战斗日志与战况摘要
- 英雄觉醒选择

## 16.3 右栏
右栏展示辅助候选：
- 节点候选
- 奖励候选
- 可替换英雄
- 当前已拥有的回响片段/遗物列表

## 16.4 原“背包/装备管理”页面的去向
如果采用轻装备方案，原有相关页面重命名/重做为：
- `构筑管理`
- `回响库`
- `遗物总览`

重点从“翻包”改成“看当前 build 组件”。

## 17. 数据模型方向

## 17.1 RunState
```yaml
run_state:
  run_id: string
  floor: int
  node_index: int
  party: [hero_instance]
  reserve: [hero_instance]
  echo_core: string
  echo_fragments: [string]
  relics: [string]
  route_history: [node_id]
  risk_contracts: [string]
```

## 17.2 HeroInstance
```yaml
hero_instance:
  hero_id: string
  current_hp: int
  awakening_choices_taken: [string]
  equipment:
    weapon: string | null
    trinket: string | null
```

## 17.3 RewardChoice
```yaml
reward_choice:
  type: echo_fragment | relic | awakening | recruit | forge
  options: [option_id]
  context_tags: [string]
```

## 18. 实现优先级建议

如果后续开始落地，推荐顺序不是“先补完旧系统”，而是：

1. 地图节点与 run state 框架
2. 战后三选一奖励流程
3. Echo Core + Echo Fragment 最小可运行集
4. 8 名首发英雄重组
5. 角色觉醒系统
6. 精英 / Boss / 风险节点
7. 极简装备（若保留）
8. 局外解锁与总结页

## 19. 明确不做的东西

以下内容至少在重设计 MVP 不做：
- 长期英雄等级树
- 账号等级
- 装备套装系统
- 复杂锻造材料链
- 大型世界地图
- 剧情驱动章节演出
- 20+ 英雄大 roster
- 9 槽位装备管理

## 20. 需要你拍板的关键问题

为了把这份 draft 收敛成正式 spec，最需要先决定的是 4 件事：

1. **是否接受 run-based 作为新的主循环**
   - 接受：项目从“长期养成”彻底转向“单局构筑”
   - 不接受：那就只能做较温和重构，而不是推翻重做

2. **装备是否降级为轻量系统**
   - A. 完全取消传统装备
   - B. 保留 2 槽极简装备

3. **局外 progression 要轻到什么程度**
   - A. 只解锁内容，不加永久数值
   - B. 少量永久开局优势

4. **首发英雄数是否锁定为 8**
   - A. 锁 8，集中打磨
   - B. 保持 10-12，丰富但更分散

## 21. 当前建议的锁定决策

如果由我先给出默认版，我建议先锁：

1. 主循环改为 **run-based 深渊远征**
2. 保留 **5 人自动战斗**
3. 保留 **CLI + 三栏 UI**
4. 删除 **长期等级作为核心玩法**
5. 删除 **大背包多槽装备作为核心玩法**

这次重设计的本质不是“把旧系统再做复杂一点”，而是：
